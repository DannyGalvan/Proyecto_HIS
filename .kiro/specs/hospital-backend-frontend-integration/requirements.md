# Requirements Document

## Introduction

This document specifies the requirements for the complete integration of the Hospital Information System (HIS) backend with the frontend client. The system is a comprehensive hospital management platform that handles patient appointments, medical consultations, laboratory orders, prescriptions, pharmacy dispensing, payments, and notifications.

The backend is built with ASP.NET Core 8.0, PostgreSQL, and JWT authentication, using a generic EntityService pattern for CRUD operations. The frontend is built with React, TypeScript, Vite, React Query, and Zod validation, following a reusable component architecture.

This integration covers 11 major modules with 24 controllers and over 50 distinct user workflows, ensuring seamless communication between frontend and backend while maintaining security, performance, and data integrity.

## Glossary

- **HIS_System**: The complete Hospital Information System including backend and frontend
- **Backend_API**: The ASP.NET Core 8.0 REST API at /api/v1/
- **Frontend_Client**: The React + TypeScript web application
- **JWT_Token**: JSON Web Token used for authentication and authorization
- **EntityService**: Generic CRUD service pattern in the backend
- **Response_Pattern**: Standard response format Response<T> with success, message, data, totalResults
- **CRUD_Controller**: Generic controller providing GetAll, GetById, Create, Update, Patch, Delete operations
- **React_Query**: Async state management library for data fetching
- **Zod_Validator**: TypeScript schema validation library
- **TableServer**: Reusable table component with pagination, filtering, and sorting
- **CatalogueSelect**: Reusable dropdown component for catalog entities
- **Operation_Sync**: Automatic synchronization of backend operations to database via reflection
- **Idempotency_Key**: UUID v4 generated by frontend to prevent duplicate operations
- **Luhn_Algorithm**: Checksum formula for validating credit card numbers
- **CIE10_Code**: International Classification of Diseases 10th revision diagnostic code
- **PCI_DSS**: Payment Card Industry Data Security Standard
- **Soft_Delete**: Logical deletion by setting State=0 instead of physical removal

### Requirement 1: Authentication — Login and Dynamic Menu

**User Story:** As a hospital system user, I want to log in with my credentials and see a navigation menu tailored to my role's permissions, so that I can access only the modules I am authorized to use.

#### Acceptance Criteria

1. WHEN a user submits valid credentials to POST /api/v1/Auth, THE Frontend_Client SHALL store the JWT_Token in localStorage under the key "@auth" and set the Authorization header for all subsequent requests.
2. WHEN the login response contains operations[], THE Frontend_Client SHALL construct the navigation menu using only modules where isVisible=true and operations where isVisible=true.
3. WHEN the login response contains redirect=true, THE Frontend_Client SHALL redirect the user to the mandatory password change page before allowing access to any other route.
4. WHEN a user submits incorrect credentials, THE Frontend_Client SHALL increment a local failure counter and display the remaining attempts.
5. WHEN a user reaches 5 consecutive failed login attempts, THE Frontend_Client SHALL display an account lockout message and disable the login form.
6. WHEN the JWT_Token expires and a protected request returns HTTP 401, THE Frontend_Client SHALL clear the stored token, redirect to the login page, and display the message "Tu sesión ha expirado vuelve a iniciar sesión".
7. WHEN a request returns HTTP 403, THE Frontend_Client SHALL display the message "No tienes permisos para realizar esta acción contacta con el administrador" without redirecting.
8. THE Frontend_Client SHALL send all audit fields (createdBy, updatedBy) as null in every request, as the Backend_API assigns them automatically from the JWT_Token.

#### Correctness Properties

- **Menu construction invariant**: FOR ALL valid login responses, the number of menu items rendered SHALL equal the count of modules in operations[] where isVisible=true. No module with isVisible=false SHALL appear in the menu.
- **Failure counter monotonicity**: FOR ALL sequences of failed login attempts, the failure counter SHALL be non-decreasing and SHALL reach exactly 5 before lockout is triggered.

### Requirement 2: Authentication — Registration, Password Management, and Token Validation

**User Story:** As an external patient, I want to register an account and manage my password, so that I can access the patient portal and recover access if I forget my credentials.

#### Acceptance Criteria

1. WHEN an external user submits the registration form to POST /api/v1/Auth/Register, THE Frontend_Client SHALL validate all fields before sending the request.
2. THE Frontend_Client SHALL reject any name field with fewer than 10 or more than 100 characters.
3. THE Frontend_Client SHALL reject any identificationDocument (DPI) that is not exactly 13 numeric digits.
4. THE Frontend_Client SHALL reject any nit that is not 8–9 alphanumeric characters.
5. THE Frontend_Client SHALL reject any phone number (number) that is not exactly 8 numeric digits.
6. THE Frontend_Client SHALL reject any password with fewer than 12 characters.
7. THE Frontend_Client SHALL reject any userName that is not 8–9 characters.
8. WHEN a user submits a recovery email to POST /api/v1/Auth/RecoveryPassword, THE Frontend_Client SHALL display a confirmation message regardless of whether the email exists, to prevent user enumeration.
9. WHEN a user accesses the password change page with a token, THE Frontend_Client SHALL validate the token via GET /api/v1/Auth/{token} before displaying the form.
10. WHEN a user submits a new password via PUT /api/v1/Auth/ChangePassword, THE Frontend_Client SHALL validate that password and confirmPassword are identical before sending the request.
11. WHEN an authenticated user resets their password via POST /api/v1/Auth/ResetPassword, THE Frontend_Client SHALL include the JWT_Token in the Authorization header.

#### Correctness Properties

- **DPI format property**: FOR ALL strings s, the DPI validator SHALL accept s if and only if s matches the pattern /^\d{13}$/.
- **NIT format property**: FOR ALL strings s, the NIT validator SHALL accept s if and only if s matches the pattern /^[a-zA-Z0-9]{8,9}$/.
- **Phone format property**: FOR ALL strings s, the phone validator SHALL accept s if and only if s matches the pattern /^\d{8}$/.
- **Password minimum length property**: FOR ALL strings s, the password validator SHALL accept s if and only if s.length >= 12.
- **Username length property**: FOR ALL strings s, the username validator SHALL accept s if and only if s.length >= 8 AND s.length <= 9.

### Requirement 3: User Management — CRUD with Extended Fields

**User Story:** As an administrator, I want to create, view, update, and deactivate user accounts with all required fields including branch assignment and insurance information, so that I can manage all hospital staff and patient accounts.

#### Acceptance Criteria

1. THE Frontend_Client SHALL display a paginated list of users using the TableServer component, fetching from GET /api/v1/User with Include=Rol.
2. WHEN creating a user, THE Frontend_Client SHALL require rolId, email, name, identificationDocument, userName, password, and state.
3. WHEN creating a user, THE Frontend_Client SHALL include the optional fields branchId, insuranceNumber, nit, and number in the UserRequest DTO.
4. WHEN updating a user, THE Frontend_Client SHALL send the user id inside the request body (not in the URL) for PUT /api/v1/User.
5. WHEN performing a partial update, THE Frontend_Client SHALL use PATCH /api/v1/User with only the fields to be modified.
6. WHEN deleting a user, THE Frontend_Client SHALL call DELETE /api/v1/User/{id} and the Backend_API SHALL set State=0 without physical deletion.
7. THE Frontend_Client SHALL display the user's rol name by including the Rol navigation property via Include=Rol.
8. WHEN a user search is performed, THE Frontend_Client SHALL return results within 3 seconds.
9. THE Frontend_Client SHALL mask the identificationDocument (DPI) and nit fields in display views, showing only the last 4 characters.
10. WHEN the insuranceNumber field is provided, THE Frontend_Client SHALL validate it is between 5 and 50 characters.

#### Correctness Properties

- **Pagination size invariant**: FOR ALL paginated requests with pageSize=N, the number of items in data[] SHALL be less than or equal to N.
- **Soft delete state invariant**: FOR ALL users after a DELETE operation, the user's state field SHALL equal 0 when retrieved via GET /api/v1/User/{id}.

### Requirement 7: Appointment Scheduling (CU-03)

**User Story:** As a patient, I want to schedule a medical appointment by selecting a specialty, branch, doctor, and date, so that I can receive medical care at a convenient time and location.

#### Acceptance Criteria

1. WHEN a patient initiates appointment scheduling, THE Frontend_Client SHALL present a multi-step form: specialty selection → branch selection → calendar with available slots → reason for visit → optional document upload → confirmation.
2. THE Frontend_Client SHALL populate the specialty dropdown from GET /api/v1/Specialty with filters State:eq:1.
3. THE Frontend_Client SHALL populate the branch dropdown from GET /api/v1/Branch with filters State:eq:1.
4. THE Frontend_Client SHALL reject any appointmentDate that is not in the future at the time of submission.
5. THE Frontend_Client SHALL reject any reason field with fewer than 10 or more than 2000 characters.
6. WHEN a patient uploads a reference document, THE Frontend_Client SHALL reject files that are not PDF format (application/pdf).
7. WHEN a patient uploads a reference document, THE Frontend_Client SHALL reject files larger than 2MB (2,097,152 bytes).
8. WHEN a patient confirms the appointment, THE Frontend_Client SHALL start a 5-minute countdown timer and display it visibly to the patient.
9. WHEN the 5-minute reservation timer expires before payment is completed, THE Frontend_Client SHALL display an expiration message and redirect the patient to restart the scheduling flow.
10. WHEN the appointment is created successfully, THE Frontend_Client SHALL redirect the patient to the payment flow.
11. WHEN scheduling a follow-up appointment (CU-11), THE Frontend_Client SHALL include followUpType and parentConsultationId in the AppointmentRequest.

#### Correctness Properties

- **Future date validation property**: FOR ALL DateTime values d, the appointment date validator SHALL accept d if and only if d is strictly greater than the current timestamp at the moment of validation.
- **Document size validation property**: FOR ALL file uploads, the validator SHALL accept the file if and only if file.size <= 2097152 AND file.type === "application/pdf".
- **Reservation timer monotonicity**: WHILE the reservation timer is active, the displayed remaining time SHALL be non-increasing and SHALL reach zero before the expiration action is triggered.

### Requirement 9: Online Payment Processing (CU-04)

**User Story:** As a patient, I want to pay for my appointment online using a credit or debit card, so that I can confirm my appointment without visiting the hospital in person.

#### Acceptance Criteria

1. THE Frontend_Client SHALL display a payment summary showing the appointment details and total amount before presenting the card form.
2. THE Frontend_Client SHALL validate the card number as 13–19 digits using the Luhn_Algorithm before submission.
3. THE Frontend_Client SHALL validate the cardholder name as 5–100 characters.
4. THE Frontend_Client SHALL validate the expiration date in MM/YY format and reject any date that is in the past.
5. THE Frontend_Client SHALL validate the CVV as 3–4 digits.
6. THE Frontend_Client SHALL never store or log the CVV value in any client-side state, localStorage, or network request body beyond the single payment submission.
7. THE Frontend_Client SHALL mask the card number input, displaying only the last 4 digits after entry.
8. THE Frontend_Client SHALL generate a UUID v4 Idempotency_Key before each payment submission and include it in the PaymentRequest.
9. WHEN the payment session exceeds 10 minutes, THE Frontend_Client SHALL invalidate the session, display a timeout message, and require the user to restart the payment flow.
10. WHEN the payment gateway returns a failure response (insufficient funds, invalid card, expired card, communication error), THE Frontend_Client SHALL display a specific error message corresponding to the failure type.
11. WHEN payment is successful, THE Frontend_Client SHALL update the appointment status to "Pagada" and display a printable receipt containing: transaction number, patient name, amount, date/time, service detail, and branch.

#### Correctness Properties

- **Luhn validation property**: FOR ALL card number strings of length 13–19 containing only digits, the Luhn_Algorithm validator SHALL return true if and only if the number passes the Luhn checksum. Modifying any single digit of a valid card number SHALL cause the validator to return false.
- **Idempotency key uniqueness property**: FOR ALL pairs of consecutive payment form submissions, the generated Idempotency_Key values SHALL be different (no two submissions share the same UUID).
- **Card masking property**: FOR ALL card number strings s of length 13–19, the masking function SHALL return a string where only the last 4 characters of s are visible and all preceding characters are replaced with a masking character.
- **CVV non-persistence property**: FOR ALL payment form submissions, the CVV value SHALL NOT be present in localStorage, sessionStorage, or any React state after the payment response is received.

### Requirement 10: Cash Payment at Reception (CU-06)

**User Story:** As a cashier, I want to process cash and POS card payments for appointments, lab orders, and pharmacy dispensing, so that patients can pay for services at the hospital reception desk.

#### Acceptance Criteria

1. THE Frontend_Client SHALL allow searching for a patient's pending payment by DPI or appointment number.
2. WHEN processing a cash payment, THE Frontend_Client SHALL require the amountReceived field and calculate changeAmount = amountReceived - amount.
3. THE Frontend_Client SHALL reject any cash payment where amountReceived is less than amount.
4. WHEN processing a POS card payment, THE Frontend_Client SHALL capture only the last 4 digits of the card number.
5. THE Frontend_Client SHALL generate a UUID v4 Idempotency_Key for every payment submission regardless of payment method.
6. WHEN payment is completed, THE Frontend_Client SHALL generate a receipt containing: transaction number, patient name, amount, date/time, service detail, and branch name.
7. THE Frontend_Client SHALL support payment for three payment types: Consulta (paymentType=0), Laboratorio (paymentType=1), and Farmacia (paymentType=2).

#### Correctness Properties

- **Change calculation property**: FOR ALL cash payments where amountReceived >= amount, changeAmount SHALL equal amountReceived - amount exactly (no rounding error beyond 2 decimal places).
- **Change non-negative invariant**: FOR ALL valid cash payments, changeAmount SHALL be greater than or equal to zero.

### Requirement 11: Vital Signs Registration (CU-07)

**User Story:** As a nurse, I want to record a patient's vital signs before their medical consultation, so that the doctor has accurate baseline health data when the consultation begins.

#### Acceptance Criteria

1. THE Frontend_Client SHALL provide a vital signs form linked to a specific appointment, sending data to POST /api/v1/VitalSign.
2. THE Frontend_Client SHALL validate bloodPressureSystolic as an integer between 60 and 250 mmHg.
3. THE Frontend_Client SHALL validate bloodPressureDiastolic as an integer between 40 and 150 mmHg.
4. THE Frontend_Client SHALL validate temperature as a decimal between 34.0 and 42.0 °C with 1 decimal place.
5. THE Frontend_Client SHALL validate weight as a decimal between 0.5 and 300.0 kg with 2 decimal places.
6. THE Frontend_Client SHALL validate height as a decimal between 30 and 250 cm with 2 decimal places.
7. THE Frontend_Client SHALL validate heartRate as an integer between 30 and 220 bpm.
8. WHEN bloodPressureSystolic is below 90 or above 140, THE Frontend_Client SHALL display a non-blocking visual alert indicating Hipotensión or Hipertensión respectively.
9. WHEN bloodPressureDiastolic is below 60 or above 90, THE Frontend_Client SHALL display a non-blocking visual alert indicating low or high diastolic pressure.
10. WHEN temperature is below 36.0, THE Frontend_Client SHALL display a non-blocking visual alert indicating Hipotermia.
11. WHEN temperature is above 37.5, THE Frontend_Client SHALL display a non-blocking visual alert indicating Fiebre.
12. WHEN heartRate is below 60, THE Frontend_Client SHALL display a non-blocking visual alert indicating Bradicardia.
13. WHEN heartRate is above 100, THE Frontend_Client SHALL display a non-blocking visual alert indicating Taquicardia.
14. THE Frontend_Client SHALL allow saving vital signs even when alerts are present — alerts SHALL NOT block form submission.
15. WHEN vital signs are saved, THE Frontend_Client SHALL reflect the saved data within 2 seconds.

#### Correctness Properties

- **Vital sign alert property**: FOR ALL VitalSignRequest values, the alert generation function SHALL produce an alert for each field that is outside its defined normal range, and SHALL produce no alert for each field that is within its normal range. The set of alerts SHALL be exactly the set of out-of-range fields.
- **Non-blocking alert invariant**: FOR ALL vital sign submissions with one or more alerts, the form submission SHALL succeed and the data SHALL be persisted. The presence of alerts SHALL NOT cause a validation error or prevent the POST request.

FOR ALL submissions where consultationStatus=0, the diagnosis field SHALL be optional.

### Requirement 13: Laboratory Order Management (CU-09)

**User Story:** As a doctor, I want to create laboratory orders from within a consultation, and as a lab technician, I want to record and publish results, so that diagnostic information flows efficiently from order to patient record.

#### Acceptance Criteria

1. WHEN a doctor creates a lab order, THE Frontend_Client SHALL send a LabOrderRequest to POST /api/v1/LabOrder with consultationId, doctorId, patientId, and orderStatus=0 (Pendiente).
2. THE Frontend_Client SHALL allow adding multiple LabOrderItem records to a lab order, each referencing a labExamId from the LabExam catalog.
3. THE Frontend_Client SHALL display pending lab orders for the lab technician via GET /api/v1/LabOrder?Filters=OrderStatus:0&Include=Patient,Items.

6. THE Frontend_Client SHALL require supervisor review before allowing a result to be published (isPublished=true).
7. WHEN results are published, THE Frontend_Client SHALL notify the doctor and the result SHALL be available in the patient's history within 2 minutes.
8. THE Frontend_Client SHALL display the total amount of a lab order as the sum of all item amounts.

#### Correctness Properties

- **Out-of-range alert property**: FOR ALL LabOrderItem records where isOutOfRange=true, the result display component SHALL render the out-of-range alert indicator. FOR ALL records where isOutOfRange=false or null, the alert indicator SHALL NOT be rendered.
- **Lab order total invariant**: FOR ALL LabOrder records with items, the displayed totalAmount SHALL equal the arithmetic sum of all LabOrderItem.amount values in that order.

### Requirement 14: Prescriptions and Pharmacy Dispensing (CU-10)

**User Story:** As a doctor, I want to create prescriptions during consultations, and as a pharmacist, I want to dispense medications with inventory tracking and substitution support, so that patients receive their prescribed medications accurately.

#### Acceptance Criteria

1. WHEN a doctor creates a prescription, THE Frontend_Client SHALL send a PrescriptionRequest to POST /api/v1/Prescription with consultationId, doctorId, and prescriptionDate.
2. THE Frontend_Client SHALL allow adding multiple PrescriptionItem records with medicineName, dosage, frequency, duration, and specialInstructions.
3. WHEN a pharmacist searches for a patient's prescriptions, THE Frontend_Client SHALL fetch from GET /api/v1/Prescription with Filters and Include=Items.
4. WHEN a pharmacist opens a prescription, THE Frontend_Client SHALL validate that the prescriptionDate is not more than 7 days before the current date.
5. WHEN a prescription is older than 7 days, THE Frontend_Client SHALL display an expiration warning and prevent dispensing.
6. FOR EACH prescription item, THE Frontend_Client SHALL check inventory availability via GET /api/v1/MedicineInventory?Filters=MedicineId:X,BranchId:Y.
7. WHEN a medication is out of stock, THE Frontend_Client SHALL offer a substitution option, requiring the pharmacist to set wasSubstituted=true and provide a substitutionReason.
8. WHEN wasSubstituted=true, THE Frontend_Client SHALL require the substitutionReason field to be non-empty.
9. WHEN dispensing is complete, THE Frontend_Client SHALL update the MedicineInventory via PATCH /api/v1/MedicineInventory with the new currentStock value.
10. WHEN currentStock becomes less than or equal to minimumStock after dispensing, THE Frontend_Client SHALL display a low-stock alert for that medication.
11. THE Frontend_Client SHALL calculate the dispense total as the sum of (quantity × unitPrice) for all DispenseItem records.
12. WHEN dispensing requires payment, THE Frontend_Client SHALL create a Payment record with paymentType=2 (Farmacia) and the dispenseId reference.

#### Correctness Properties

- **Prescription validity property**: FOR ALL Prescription records, the Frontend_Client SHALL allow dispensing if and only if the difference between the current date and prescriptionDate is less than or equal to 7 days.
- **Dispense total invariant**: FOR ALL Dispense records, the displayed totalAmount SHALL equal the sum of (DispenseItem.quantity × DispenseItem.unitPrice) for all items in that dispense.
- **Substitution reason invariant**: FOR ALL DispenseItem records where wasSubstituted=true, the substitutionReason field SHALL be non-null and non-empty. FOR ALL records where wasSubstituted=false or null, substitutionReason SHALL be optional.
- **Low-stock alert property**: FOR ALL MedicineInventory records after a dispense operation, the Frontend_Client SHALL display a low-stock alert if and only if currentStock <= Medicine.minimumStock.

### Requirement 17: API Communication and Response Pattern

**User Story:** As a developer, I want all frontend-backend communication to follow a consistent pattern with proper error handling, so that the application behaves predictably and users receive meaningful feedback for all operations.

#### Acceptance Criteria

1. THE Frontend_Client SHALL use React_Query for all asynchronous data fetching and mutation operations.
2. THE Frontend_Client SHALL use Zod_Validator schemas for all form validations before submitting data to the Backend_API.
3. THE Frontend_Client SHALL follow the Response_Pattern: every API response SHALL be typed as ApiResponse<T> with fields success, message, data, and totalResults.
4. WHEN success=false in an API response, THE Frontend_Client SHALL display the message field as a toast notification.
5. WHEN success=false and data contains an array of validation failures, THE Frontend_Client SHALL map each failure to its corresponding form field error.
6. THE Frontend_Client SHALL use the existing axios interceptors for all requests, which automatically attach the JWT_Token from localStorage.
7. ALL list endpoints SHALL support the query parameters: Filters, Include, PageNumber (default 1), PageSize (default 30), and IncludeTotal.
8. WHEN sending dates to the Backend_API, THE Frontend_Client SHALL use ISO 8601 format (yyyy-MM-ddTHH:mm:ss).
9. WHEN displaying dates from the Backend_API, THE Frontend_Client SHALL parse the dd/MM/yyyy HH:mm:ss format returned by the backend.
10. THE Frontend_Client SHALL use the existing service pattern (authService, catalogueService, operationService) as the template for all new service modules.
11. THE Frontend_Client SHALL use the TableServer component for all paginated list views.
12. THE Frontend_Client SHALL use the CatalogueSelect component for all foreign-key dropdown fields.

#### Correctness Properties

- **Response pattern round-trip property**: FOR ALL ApiResponse<T> objects received from the Backend_API, deserializing the JSON SHALL produce an object where success is a boolean, message is a string, data is of type T or T[], and totalResults is a number. No field SHALL be lost or mutated during deserialization.
- **Pagination consistency property**: FOR ALL paginated list responses with IncludeTotal=true, the totalResults field SHALL be greater than or equal to the count of items in data[]. FOR ALL pages beyond the first, the sum of items across all pages SHALL equal totalResults.

### Requirement 18: Security and Compliance

**User Story:** As a hospital IT administrator, I want the frontend to enforce security standards for authentication, payment data, and sensitive personal information, so that the system complies with PCI DSS and protects patient privacy.

#### Acceptance Criteria

1. THE Frontend_Client SHALL communicate with the Backend_API exclusively over HTTPS (TLS 1.2 or higher).
2. THE Frontend_Client SHALL include the JWT_Token in the Authorization header as "Bearer {token}" for all authenticated requests.
3. THE Frontend_Client SHALL never store the full card number, CVV, or expiration date in localStorage, sessionStorage, or any persistent client-side storage.
4. THE Frontend_Client SHALL generate a new UUID v4 Idempotency_Key for every payment submission to prevent duplicate charges.
5. THE Frontend_Client SHALL mask DPI and NIT values in all display views, showing only the last 4 characters.
6. WHEN a payment session exceeds 10 minutes of inactivity, THE Frontend_Client SHALL clear the payment form and require the user to restart.
7. WHEN a temporary appointment reservation exceeds 5 minutes, THE Frontend_Client SHALL release the reservation and notify the user.
8. THE Frontend_Client SHALL apply PCI_DSS-compliant input handling for all payment forms: no autocomplete on card fields, masked input display, and no clipboard access to card data.

#### Correctness Properties

- **Idempotency key format property**: FOR ALL generated Idempotency_Key values, the key SHALL match the UUID v4 format pattern /^[0-9a-f]{8}-[0-9a-f]{4}-4[0-9a-f]{3}-[89ab][0-9a-f]{3}-[0-9a-f]{12}$/i.
- **Card masking completeness property**: FOR ALL card number strings s of length L (13 <= L <= 19), the masking function SHALL produce a string of the same length L where the first (L-4) characters are replaced with a masking character and the last 4 characters are identical to the last 4 characters of s.

### Requirement 19: Performance Requirements

**User Story:** As a hospital user, I want the application to respond quickly to my interactions, so that clinical workflows are not delayed by slow software.

#### Acceptance Criteria

1. WHEN a user performs a search or filter operation, THE Frontend_Client SHALL display results within 3 seconds.
2. WHEN a doctor opens a patient's medical record, THE Frontend_Client SHALL load all required data within 2 seconds.
3. WHEN a page is navigated to, THE Frontend_Client SHALL complete the initial render within 3 seconds.
4. WHEN a user types in the CIE-10 autocomplete field, THE Frontend_Client SHALL display suggestions within 500ms.
5. WHEN vital signs are saved, THE Frontend_Client SHALL reflect the updated data within 2 seconds.
6. WHEN lab results are published, THE Frontend_Client SHALL make them available in the patient's history within 2 minutes.
7. WHEN a POS terminal payment is processed, THE Frontend_Client SHALL receive a response within 15 seconds.
